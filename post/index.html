CVE-2021-4034 (PWNKIT - Linux Privilege Escalation)

This is a local linux privillege escalation in polkit’s pkexec that affects almost every linux distribution. That means that any user can become root when they exploit this vulnerability. Today we are going to look at the effects of OOB (out-of-bounds) read and write and how they can be critical.

  • This was a vulnerablity that was discovered by the security company Qualys

Recap

  1. What is an OOB vulneraility?
  • An out-of-bounds write vulnerability this is when a process or a software writes data past the end or before the beginning of the intended buffer.
  • An out-of-bounds read vulnerability this is when a process reads data from outside the bounds of allocated memory.Therefore both of these can lead to memory corruption, a crash or RCE.We are going to see in play the effect of both these vulnerabilities that leads to priv escalation.
  1. What is this polkit pkexec in linux?

  • This is a tool from the polkit or policy kit software package and it is a common component of modern unix-like systems. It is used to control system-wide privilleges and provides an organized way for non-privilleged processes to communicate with privilleged processes.

  • According to the linux manual page

man polkit

PolicyKit provides an authorization API intended to be used by privileged 
programs (“MECHANISMS”) offering service to unprivileged programs (“CLIENTS”) 
through some form of IPC mechanism such as D-Bus or Unix pipes.

man pkexec

pkexec allows an authorized user to execute a PROGRAM as another user and 
If that username is not specified, then the program will be executed as 
the admin super user that is root.

Vulnerability

  • As started earlier the vulnerability is an out-of-bounds read and write vulnerability and this can be seen from the source code snippet of pkexec that was published by Qualys Security Company during their disclosure.
  1. The main function that processes the command-line arguments. (pkexec takes in different arguments from the command-line)
int main(int argc, char *argv[]){
	....
	for(n=1, n < (guint) argc, n++){
		// do something
	}
	
	path = g_strdup(argv[n]);
	
	if (path[0] != "/"){
		s = g_find_program_in_path(path);
		argv[n] = path = s;
	}
}

  • From the above code, when we pass zero arguments to the program when running /usr/bin/pkexec, argc will be 0 and argv[0] - will be NULL.

  • Therefore from the above snippet, the followig happens.

    1. The value of (n) was initialized to (1).
    2. The pointer (path) points to the malloced() address of argv[1] and this is where we have out OOB-read vulnerability.
    3. The pointer (s) writes into argv[1] again and this leads to an OOB-write vulnerability.

  • Since we did not provide any arguments in the command-line, what is read and written into these pointers?. When a new process is called with the execve() syscall or when you run a new process from the terminal with command-line arguments, the kernel copies out arguments and all eviroment variable strings and pointers to the end of the new programs stack. That is to the end of the main’s function stack.

  • Therefore argv and envp pointers are contagious in memory in that they are next to each other in memory as shown in the image below.

  • Therefore this means that when we pass no argument to pkexec, argv[1] now points to out first enviroment variable that is set. Let us assume the following scenarion:
    1. We add an enviroment variable called “value”.
    2. When calling pkexec with not command-line arguments, the code snippet above will allocate argv[1] that is now “value” dynamically on the heap and the address returned to (path).
    3. It will then check if path[0] begins with “/” to determine if we if its an absolute path or not.
    4. Since “value” does not contain “/” it will execute g_find_program_in_path(path) and the value returned to “s” will the absolute path of “value” and this will be used to overwrite argv[1] which is now envp[0] (our environment variable).

Exploitation

  • How can this be exploited to out gain root? According to Qualys Security team, Since we control envp[0], that is an enviroment variable using the OOB-write vulnerability, we can write anything to this enviroment variable. Therefore we need to look at some of the dangerous enviroment variables that we can exploit.

(Note that after the out-of-bouds write above, the process clears out its enviroment variables.)

  1. LD_PRELOAD:
  • This is an enviroment variable that specifies the first shared lib to load. We can therefore use the OOB-write to write this into our env variables but unfortuanetly, (ld.so) clears these types of variables out when running suid processes and before calling the main() function during execution.
  1. GCONV_PATH

  • This is an enviroment variable somewhat similar to LD_LIBARY_PATH in functionality and this is part of GLib that is the GNOME Library. GLib is a library that is build on top of the C standard library and provides functions like error logging,authentication and PATH lookups.

  • Therefore since certain functions return messages to users, encoding and decoding to and fro different formats must be considered and this is where GCONV_PATH comes into play.

  • When a process encounters e.g “CHARSET=UTF-8” enviroment variable, GLib loads $GCONV_PATH/gconv-modules or when this variable is not set the lookup is done from ("/usr/lib/gconv/gconv-modules"). This is a file that contains something like “BS_4730// UTF-8 ISO646 2” and this is used to define a module located at “ISO646.so” that will allow “BS_4730” to be re-encoded as “UTF-8”. (this is just a simple example).

  • Therefore when we control the GCONV_PATH, we can therefore cause this SUID process to run with full root privilleges by loading a lib that we control in the share “.so” file.

Proove Of Concept

  • Below I have provided a snippet of my POC and the full payload can be found at pwnkitty.c there are also other different POC payloads out there with the same functionality so be sure to check them out.
int main(int argc, char **argv){
	char *pkexec_envp[] = {"tmp", "CHARSET=xxx", "PATH=GCONV_PATH=."};
	char *pkexec_args[] = {NULL};
	
	mkdir("GCONV_PATH=.", 0755);
	open("GCONV_PATH=./tmp", O_CREAT, 0755);
	mkdir("tmp", 07555);
	open("tmp/gconv-modules", O_WRONLY, 0755);

	execve("/usr/bin/pkexec", pkexec_args, pkexec_envp);
}
  • The above snippet works in the following way.

    1. Since we provided no arguments when calling the “pkexec”, argv[1] now becomes whatever we loaded in out enviroment variable and that is “tmp”.

    2. Since “tmp” does not start with “/”, the process will look for its absolute path and it will check from the paths provided by the PATH enviroment variable.

    3. Since “PATH” variable contains our spoofed directory “GCONV_PATH=.” the value that will be returned by g_find_program_in_path() will be “GCONV_PATH=./tmp”.

    4. “GCONV_PATH=./tmp” this is a valid environment variable therefore using the oob-write we overwrite into envp[0] = argv[1] = “GCONV_PATH=./tmp” therefore since we also included a “CHARSET=xxx” GLib therefore now comes into play and looks for “GCONV_PATH” from the enviroment variables that contains “./tmp” as its value.

    5. It will then go to out “./tmp” directory to look for “gconv-modules” and based on the contents, it will load a library from “./tmp” that we control and when its loaded this will lead to our privilege escalation.

Mitigations

  1. Update pkexec to the latest version.
  2. Remove the SUID bit in the executable.
